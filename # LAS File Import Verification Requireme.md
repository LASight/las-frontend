# LAS File Import Verification Requirements (Using lasio)

## 1. File Loading and Parser Behavior

The system must:
- Use `lasio.read()` as the primary ingestion method.
- Capture and surface parsing warnings generated by lasio (e.g., malformed lines, ignored content).
- Detect when lasio applied fallback parsing strategies (e.g., flexible delimiter handling).

The system must allow the user to:
- Proceed with a partially parsed file.
- Be informed when the parser made assumptions.

The system should:
- Avoid re-implementing low-level parsing already handled by lasio.

---

## 2. Section and Structure Validation

The system must:
- Validate sections based on lasio’s parsed object:
  - `las.version`
  - `las.well`
  - `las.curves`
  - `las.data`
- Detect missing or empty sections after parsing.

The system must allow the user to:
- Proceed if non-critical sections are missing.

The system should:
- Trust lasio’s structural interpretation rather than re-validating raw sections.

---

## 3. Header–Data Consistency

The system must:
- Compare `len(las.curves)` with `las.data.shape[1]`.
- Detect duplicate or empty curve mnemonics.

The system must allow the user to:
- Rename curves.
- Reassign column meanings if mismatch exists.
- Drop extra or invalid columns.

The system should:
- Not assume lasio’s curve mapping is always correct.

---

## 4. Depth Handling

The system must:
- Use `las.index` (or first curve) as the default depth reference.
- Validate numeric values and general monotonicity.
- Extract depth unit from curve metadata when available.

The system must allow the user to:
- Override the depth curve.
- Override depth units.
- Optionally enable sorting or cleaning.

The system should:
- Avoid automatic modification without user confirmation.

---

## 5. Missing / NULL Values

The system must:
- Read NULL value from `las.well.NULL` if present.
- Detect if lasio has already converted NULLs to `NaN`.
- Estimate missing value percentage per curve.

The system must allow the user to:
- Confirm or override the global NULL value.
- Define a small set of additional NULL values if needed.

The system should:
- Operate primarily using `NaN` internally.
- Avoid per-value validation or prompts.

---

## 6. Units and Curve Metadata

The system must:
- Extract units and descriptions from `las.curves`.
- Detect missing or empty units.

The system must allow the user to:
- Edit units and descriptions.
- Optionally standardize units.

The system should:
- Treat metadata as best-effort, not authoritative.

---

## 7. Curve Data Quality Checks

The system must:
- Compute basic statistics per curve:
  - min, max, mean
  - percentage of missing values
- Detect:
  - constant or near-constant curves
  - obvious outliers (threshold-based)

The system must allow the user to:
- Exclude curves from further processing.

The system should:
- Present aggregated warnings instead of per-point issues.

---

## 8. Curve Naming and Semantic Mapping

The system must:
- Detect duplicate or empty curve names.

The system must allow the user to:
- Rename curves.
- Optionally map to standardized naming.

The system should:
- Provide suggestions without enforcing naming standards.

---

## 9. Well Metadata Validation

The system must:
- Extract metadata from `las.well`.
- Detect missing or empty key fields.

The system must allow the user to:
- Edit or complete metadata.

The system should:
- Treat metadata as optional for processing.

---

## 10. Multi-Run and Overlapping Data Detection

The system must:
- Detect overlapping or non-monotonic depth intervals as indicators of multiple runs.
- Identify duplicated curve sets.

The system must allow the user to:
- Proceed with raw data.
- Optionally apply simple cleanup (e.g., sorting by depth).

The system should:
- Avoid complex automatic merging logic.

---

## 11. Post-Import User Validation Modal

The system must:
- Display a modal card immediately after parsing.
- Present a structured summary including:
  - LAS version
  - Number of curves
  - Depth range and selected depth curve
  - NULL value
  - Curve list with units and statistics
  - Key metadata
  - Parsing warnings and assumptions

- Highlight detected issues:
  - Header–data mismatches
  - Missing units
  - High missing value percentages
  - Non-monotonic depth
  - Duplicate or empty curve names

The system must allow the user to:
- Confirm and proceed, or
- Modify the interpreted schema within the modal:
  - Rename/exclude curves
  - Edit units
  - Override NULL value
  - Select depth curve
  - Edit metadata

- Apply changes and re-validate before continuing.

The system should:
- Organize content into sections (Summary, Curves, Warnings, Metadata).
- Prioritize aggregated insights over raw data.
- Support inline editing.

The system must not:
- Allow progression without explicit user confirmation.
- Use multiple dialogs; validation must occur in a single modal.
